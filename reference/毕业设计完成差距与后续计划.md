# 毕业设计完成差距与后续计划

## 一、当前已完成内容

根据《任务书》和《开题报告》的要求，毕业设计目标是完成“基于轻量化联邦学习的物联网恶意流量检测系统”，核心任务包括轻量化检测模型设计、高效联邦学习框架设计、物联网场景下的性能验证。结合当前项目目录与已有结果，项目已经具备以下基础：

### 1. 数据预处理与客户端划分

- 已有 `scripts/data_preprocess.py`、`scripts/data_preprocess_sampling_style.py`、`scripts/repack_processed_to_dataset_layout.py` 等数据处理脚本。
- 已生成多组联邦学习客户端数据目录，包括：
  - `dataset/IoT`
  - `dataset/IoT_20k_c20`
  - `dataset/IoT_20k_c20_noniid`
  - `dataset/IoT_k40_c20_noniid`
- 已存在 `data/processed_data*` 系列中间数据，说明已经完成原始 CSV 流量数据到客户端训练/测试数据的转换。
- 已支持多客户端划分，当前主要实验配置包括 20 个客户端、Non-IID 分布、少样本扩展等。

### 2. 联邦训练框架

- `src` 目录中已经实现主要联邦学习流程，包含服务器端、客户端、模型、工具函数等模块。
- 当前支持三类训练/对比算法：
  - `Local`：客户端本地独立训练。
  - `FedAvg`：传统参数平均联邦学习基线。
  - `FedProto`：基于类别原型聚合的联邦原型学习方法。
- FedProto 已实现本地原型提取、服务端类别原型加权聚合、全局原型下发与本地原型正则约束。
- 已支持早停、评估间隔、客户端参与比例、客户端数量、模型族选择等实验参数。

### 3. 轻量化与扩展模型

- 已实现面向 77 维流量特征的模型族：
  - `IoT_MLP`
  - `IoT_CNN1D`
  - `IoT_Transformer1D`
  - `IoT_MIX_MLP_CNN_TRANS`
- 模型均提供 `extract_features()` 接口，能够与 FedProto 的原型提取流程对接。
- 已有 `scripts/report_iot_model_params.py` 用于统计不同模型参数量，具备轻量化分析基础。

### 4. 实验与结果输出

- `results` 目录中已有多轮 Local、FedAvg、FedProto 的实验日志与 `.h5` 结果文件。
- 已有指标包括：
  - Accuracy
  - AUC Macro
  - AUC Micro
  - FNR
  - 训练耗时或平均轮次耗时
- 已有少样本 CNN / Transformer 实验计划与部分 Transformer 实验日志。
- 已具备论文实验表格的数据基础，但仍需要统一整理、筛选和解释。

## 二、距离完成毕业设计还差哪些内容

### 1. 数据集叙述与论文口径仍需统一

当前 `data/raw_data` 中的文件名包括 `Friday-WorkingHours-Afternoon-DDos.pcap_ISCX.csv`、`Monday-WorkingHours.pcap_ISCX.csv` 等，更接近 CICIDS2017 风格的网络流量 CSV。开题报告中多处提到 N-BaIoT、IoT botnet、物联网恶意流量检测等表述。

后续必须明确最终论文采用的数据集口径：

- 如果继续使用当前 CSV 数据，应在论文中说明其作为网络入侵/恶意流量检测数据集，用于模拟物联网边缘节点恶意流量识别场景。
- 如果论文坚持 N-BaIoT 或 IoT botnet 数据集表述，则需要补充对应数据来源、预处理脚本和实验结果。
- 不建议在最终论文中混用“当前实际数据集”和“开题预期数据集”而不解释，否则答辩时容易被追问。

### 2. 主实验矩阵尚未最终定稿

已有多批结果，但目前还缺少一个明确的“最终论文主实验矩阵”。需要确定：

- 主数据集使用 `IoT_20k_c20_noniid` 还是 `IoT_k40_c20_noniid`。
- 主模型使用 `IoT_MLP`，还是将 `IoT_CNN1D` / `IoT_Transformer1D` 作为扩展实验。
- 主对比算法是否固定为 `Local`、`FedAvg`、`FedProto`。
- 是否单独设置异构客户端实验，即使用 `IoT_MIX_MLP_CNN_TRANS` 验证 FedProto 对模型异构的兼容性。
- 是否补充消融实验，如不同 `lamda`、不同 `feature_dim`、不同 Non-IID 程度、不同少样本规模。

### 3. 结果整理与论文图表不足

当前结果分散在多级目录和 `.out` / `.h5` 文件中，尚未形成论文可直接引用的统一材料。需要补齐：

- 主实验结果汇总表。
- 不同模型参数量对比表。
- FedProto 与 FedAvg 的通信量估算表。
- Accuracy / AUC / FNR 曲线图。
- 少样本场景下 Local、FedAvg、FedProto 的对比图。
- FedProto 原型分布或特征分布可视化图。
- 对异常结果的解释，例如某些实验中 Local 明显优于 FedProto，或 FedProto 在小样本设置下准确率不稳定。

### 4. 轻量化论证还不完整

开题报告强调“双重轻量化”，即本地模型轻量化和通信轻量化。目前代码已经具备模型参数量统计与 FedProto 原型通信机制，但论文还需要更充分的量化支撑：

- 模型参数量：MLP、CNN1D、Transformer1D 参数量对比。
- 通信量：FedAvg 每轮上传/下发模型参数量，FedProto 每轮上传/下发类别原型量。
- 计算量：可补充 FLOPs、单轮训练耗时、单次推理耗时。
- 存储开销：客户端模型大小、服务端全局原型大小。

### 5. 工程交付文档不足

当前 `README.md` 为空，不利于毕业设计验收和答辩展示。至少需要补齐：

- 项目简介。
- 目录结构说明。
- 环境依赖与安装命令。
- 数据预处理流程。
- Local / FedAvg / FedProto 运行命令。
- 少样本实验和 Transformer 实验运行命令。
- 结果文件说明。
- 常见问题说明，如 CUDA 不可用、数据路径不一致、Windows / Linux 脚本差异。

### 6. 代码目录与临时文件需要整理

当前同时存在 `src` 和 `sysytem` 两套相似代码目录，容易造成最终交付口径混乱。需要明确：

- `src` 作为最终主代码目录。
- `sysytem` 作为旧版代码、备份代码或参考代码，不作为最终验收入口。
- `debug-bfc949.log`、`temp`、`tensorboard` 等临时文件是否保留，需要在 README 或交付说明中解释。
- `dataset.zip` 是否作为交付数据包保留，需要说明其内容和生成方式。

### 7. 论文正文仍需补齐工程实现与实验分析

毕业论文至少还需要完善以下内容：

- 系统总体架构图。
- FedProto 在本项目中的算法流程图。
- 数据预处理与客户端划分说明。
- 模型结构说明。
- 实验设置表。
- 实验结果与对比分析。
- 轻量化效果分析。
- 非 IID 与少样本场景分析。
- 总结与不足。

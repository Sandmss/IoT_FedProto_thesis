# 本周执行清单

## 本周目标

本周目标不是优先写论文，而是用一周时间尽可能完成一个可运行、可复现、可展示结果的“基于轻量化联邦学习的恶意流量检测系统”。

本周结束前，希望至少完成以下 5 项：

- 完成系统主线代码收口，明确 `src` 为最终运行入口
- 跑完核心实验矩阵：`Local / FedAvg / FedProto`
- 至少完成 `MLP / CNN1D / Transformer1D` 三种模型中的主要实验
- 补齐轻量化分析：参数量、通信量、耗时
- 形成一套可直接展示的结果表和运行说明

## 本周工作原则

- 先完成系统主线，再做扩展项
- 先保证实验能跑通，再追求实验更全面
- 先做“可交付版本”，再做“更漂亮版本”
- 异构模型是扩展项，不阻塞主系统完成

## 最终要交付的系统版本

本周系统完成标准建议定为：

- 数据输入：已有 IoT 恶意流量处理后数据可直接加载
- 训练算法：支持 `Local`、`FedAvg`、`FedProto`
- 模型选择：支持 `IoT_MLP`、`IoT_CNN1D`、`IoT_Transformer1D`
- 结果输出：支持保存 `.out` 日志和 `.h5` 结果文件
- 效率分析：支持参数量、通信量、训练耗时统计
- 运行入口：用 `src/main.py` 和 `src/run_*.sh`

## 本周优先级

### 最高优先级

1. 跑通并整理主实验
2. 收口系统入口与目录说明
3. 整理结果表
4. 补运行说明

### 中优先级

1. 补轻量化分析
2. 整理图表
3. 跑扩展模型实验

### 低优先级

1. 正式异构模型实验
2. 剪枝和量化实现
3. 更多消融实验

## Day 1：系统收口与实验矩阵确定

### 任务

- 确认最终主代码目录使用 `src`
- 确认 `sysytem` 不再作为主要实验入口
- 确认主数据集优先使用当前已经稳定可跑的一套
- 确认本周主实验矩阵

### 建议实验矩阵

- 算法：
  - `Local`
  - `FedAvg`
  - `FedProto`
- 模型：
  - `IoT_MLP`
  - `IoT_CNN1D`
  - `IoT_Transformer1D`
- 数据设置：
  - 优先 `n=4`
  - 有余力再补 `n=6`

### 当天产出

- 一份固定的实验矩阵
- 一份结果记录表模板
- 一份“本周只做什么、不做什么”的范围说明

## Day 2：完成 CNN1D 主实验

### 任务

- 跑 `CNN1D + Local`
- 跑 `CNN1D + FedAvg`
- 跑 `CNN1D + FedProto`
- 优先在一个代表性设置下完成，例如 `n=4`

### 检查项

- 日志文件是否正常生成
- `.h5` 结果是否正常保存
- 指标是否完整：
  - Accuracy
  - AUC Macro
  - AUC Micro
  - F1
  - FNR
  - FPR
  - 耗时

### 当天产出

- 一组完整的 `CNN1D` 三算法对比结果
- 第一版结果汇总记录

## Day 3：完成 MLP 或 Transformer 主实验补齐

### 任务

- 如果 `Transformer` 结果已经可用，就优先补 `MLP`
- 如果 `MLP` 已经部分可用，就检查并整理现有结果
- 最终目标是至少有两种模型完成完整三算法对比

### 推荐顺序

1. 先确保 `CNN1D`
2. 再补 `Transformer`
3. 最后补 `MLP`

### 当天产出

- 至少两类模型的完整实验结果
- 一张主实验总表初稿

## Day 4：补第三种模型与轻量化统计

### 任务

- 补完第三种模型的主实验
- 跑或整理参数量统计
- 跑或整理通信量统计
- 记录平均每轮耗时、总耗时、推理延迟

### 建议使用的现有脚本

- `scripts/report_iot_model_params.py`
- `scripts/report_iot_efficiency.py`

### 当天产出

- 三种模型的完整结果
- 模型参数量对比表
- 通信量对比表
- 训练耗时统计表

## Day 5：系统展示版本整理

### 任务

- 整理 `results` 目录
- 区分不同模型结果目录
- 整理一份最终结果总表
- 整理关键结论

### 最少要有的表格

- 主实验对比表
- 模型轻量化对比表
- 通信开销对比表

### 最少要有的图

- Accuracy 曲线
- AUC 曲线
- FNR 曲线

### 当天产出

- 一套可展示的结果材料
- 一套整洁的结果目录

## Day 6：补运行说明与可复现入口

### 任务

- 补 `README.md`
- 说明系统目标、目录结构、依赖环境
- 说明如何运行：
  - `Local`
  - `FedAvg`
  - `FedProto`
- 说明如何切换模型：
  - `IoT_MLP`
  - `IoT_CNN1D`
  - `IoT_Transformer1D`
- 说明结果文件在哪

### 最少写清楚

- 环境安装
- 数据准备
- 训练命令
- 结果输出说明
- 当前系统支持范围

### 当天产出

- 一份可复现系统的 `README`
- 一组最小可运行命令

## Day 7：扩展项冲刺与最终验收

### 主任务

- 从头检查系统是否能按预期运行
- 检查所有实验结果是否已记录
- 检查每个模型的结果是否都能追溯到日志文件

### 如果主线已经完成，可选补充

- 尝试 `IoT_MIX_MLP_CNN_TRANS + FedProto`
- 做一个最小异构实验
- 只要能跑通并出结果即可，不必追求很多组

### 最终验收标准

- `src/main.py` 能稳定运行
- `Local / FedAvg / FedProto` 能跑
- 至少两种以上模型完成完整实验
- 有完整结果表
- 有轻量化指标表
- 有运行说明

## 本周最低完成线

如果时间非常紧，至少完成以下内容：

- 完成 `CNN1D + Local / FedAvg / FedProto`
- 保留并整理 `Transformer` 现有结果
- 补一组 `MLP` 或至少给出参数量统计
- 完成一张主实验总表
- 完成一张轻量化/通信量表
- 完成 `README.md`

## 建议结果记录格式

建议你单独维护一个总表，字段如下：

- 数据集
- 设置
- 模型
- 算法
- Best Acc
- AUC Macro
- AUC Micro
- Precision
- Recall
- F1
- FNR
- FPR
- 平均每轮耗时
- 总耗时
- 通信量
- 日志文件路径

## 本周不建议优先做的事

- 不优先做论文精修
- 不优先做剪枝或量化代码实现
- 不优先做很多超参数消融
- 不优先做复杂异构联邦平均

这些工作都可能有价值，但会明显拖慢系统主线完成。

## 每天结束前检查

- 今天新增实验是否已经记录到总表
- 今天跑出的结果是否已经保存路径
- 当前主线是否仍然是 `Local / FedAvg / FedProto`
- 现在最阻塞系统完成的 1 个问题是什么

## 本周一句话路线

本周的正确节奏是：  
先收口系统入口，再跑完三类模型的核心实验，补轻量化指标，整理结果与运行说明，最后有余力再尝试异构扩展。

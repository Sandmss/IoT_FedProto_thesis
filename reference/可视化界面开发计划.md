# 可视化界面开发计划

## 1. 目标定位

根据导师要求，毕业设计最终系统需要提供一个本地可运行的可视化界面，并支持以下完整链路：

1. 在功能界面选定客户端，客户端以本地分文件夹形式体现，数据集存放在对应文件夹内
2. 在功能界面运行训练，并进行模型存储
3. 在功能界面加载训练好的模型，选定测试集，展示测试结果

结合你已经确定的技术路线，最终系统定位为：

- 一个本地桌面应用
- 一个基于 `Electron + Node.js + TypeScript` 的可视化联邦学习原型系统
- 一个封装现有 Python 联邦训练代码的上层控制界面

这个系统的核心不是“重写训练算法”，而是“把现有联邦训练、模型保存、结果读取、测试分析流程，用桌面软件界面的方式组织起来”。

## 2. 为什么采用 Electron + Node.js + TypeScript

### 2.1 采用原因

相较于浏览器式原型界面，`Electron` 更符合你和导师当前对“本地化可视化界面”的理解：

- 有独立窗口，更像传统软件
- 能直接访问本地文件系统
- 适合做目录选择、模型选择、测试集选择
- 能方便地调用本地 Python 脚本
- 答辩展示时更容易被理解为“系统”

### 2.2 TypeScript 的优势

采用 `TypeScript` 的好处：

- 代码结构更清晰
- 主进程、渲染进程、接口定义更容易维护
- 后续文件路径、任务状态、结果结构更容易做类型约束
- 比纯 JavaScript 更适合中等规模毕设项目

## 3. 总体系统架构

建议将最终系统划分为四层：

### 3.1 桌面界面层

由 `Electron Renderer` 实现，负责：

- 显示页面
- 接收用户选择
- 展示训练状态
- 展示测试结果与图表

### 3.2 桌面调度层

由 `Electron Main Process` 实现，负责：

- 管理本地窗口
- 选择本地目录和文件
- 调用 Python 脚本
- 管理训练/测试进程
- 把日志和结果传回界面

### 3.3 Python 训练与测试层

复用当前项目的 Python 代码，负责：

- 数据读取
- 客户端训练
- 联邦聚合
- 模型保存
- 测试评估
- `.h5` 结果输出

### 3.4 本地资源层

负责组织：

- `dataset/` 下的数据目录
- `results/` 下的实验结果目录
- 模型权重保存目录
- 日志文件目录

## 4. 功能需求拆解

## 4.1 功能一：客户端与数据集选择

导师要求：

- 在功能界面选定客户端
- 客户端可以以分文件夹形式本地体现
- 数据集分放在对应文件夹内

据此，系统需要支持：

- 选择数据集根目录
- 读取本地客户端目录结构
- 展示训练集和测试集目录
- 支持选择全部客户端或部分客户端
- 让用户明确看到当前实验使用的是哪一批数据

建议界面展示内容：

- 数据集根目录路径
- 可用客户端数量
- 训练集目录
- 测试集目录
- 已选择客户端列表

### 4.2 功能二：训练运行与模型存储

导师要求：

- 在功能界面运行训练
- 进行模型存储

据此，系统需要支持：

- 选择模型类型
- 选择算法类型
- 配置训练参数
- 点击按钮后启动训练
- 实时显示训练日志
- 训练完成后显示模型保存路径和结果文件路径

建议训练配置项：

- 数据集
- 算法：`Local / FedAvg / FedProto`
- 模型：`IoT_MLP / IoT_CNN1D / IoT_Transformer1D / IoT_MIX_MLP_CNN1D`
- 客户端数量
- 全局轮数
- 本地 epoch
- 学习率
- join ratio
- feature dimension
- lambda
- early stop patience

建议训练输出项：

- 当前状态：未开始 / 训练中 / 成功 / 失败
- 日志实时输出
- 训练耗时
- 最佳指标摘要
- 模型保存路径
- `.h5` 结果文件路径

### 4.3 功能三：模型加载、测试集选择与结果展示

导师要求：

- 在功能界面加载训练好的模型
- 选定测试集
- 展示测试结果

据此，系统需要支持：

- 选择已有实验目录或已有模型文件
- 选择测试集目录
- 执行测试或读取现有测试结果
- 展示指标与图形

建议展示指标：

- Accuracy
- AUC Macro
- AUC Micro
- Precision
- Recall
- F1
- FNR
- FPR
- 推理时延

建议展示图形：

- Accuracy 曲线
- AUC 曲线
- FNR / FPR 曲线
- 混淆矩阵
- 原型分布图
- t-SNE 图

## 6. 技术实现方案

## 6.1 前端界面

建议技术栈：

- `Electron`
- `TypeScript`
- `React` 或 `Vue`

若你想尽量稳妥，建议：

- `Electron + React + TypeScript`

原因：

- 资料多
- 组件生态成熟
- 表单、表格、图表更容易组织

如果你想更轻量，也可以：

- `Electron + 原生 HTML/CSS + TypeScript`

但从中后期维护体验看，还是建议至少用一个成熟前端框架。

## 6.2 主进程与 Python 集成

建议由 `Electron Main Process` 负责调用 Python。

推荐方式：

- 使用 `child_process.spawn` 启动 Python 命令
- 实时读取标准输出和标准错误
- 把日志通过 IPC 发送到前端界面

不建议：

- 在前端直接调用 Python
- 把训练逻辑迁移到 Node.js

原则是：

- 训练仍由 Python 执行
- Electron 只做调度与展示

## 6.3 结果读取方案

结果读取建议分两种：

### 方式一：直接读取现有 `.h5`

由 Python 端提供一个轻量脚本，把 `.h5` 中关键指标整理为 JSON，再传给 Electron。

优点：

- 复用现有结果
- 改动小

### 方式二：训练结束时同步导出 JSON

在现有训练完成后，额外导出一份 JSON 结果摘要，供界面直接读取。

优点：

- Electron 端读取更方便
- 历史结果展示更轻量

建议优先采用：

- 训练继续保留 `.h5`
- 同时补一份 JSON 摘要文件

## 6.4 本地目录选择

老师明确强调本地文件夹形式，因此目录选择是核心交互。

建议使用：

- `dialog.showOpenDialog`

支持选择：

- 数据集目录
- 模型目录
- 测试集目录
- 历史实验目录

## 10. 最小可验收版本定义

第一版可验收系统建议达到以下程度：

- 可以在桌面界面中选择数据集目录
- 可以选择客户端
- 可以选择算法和模型
- 可以启动训练
- 可以看到训练日志
- 可以确认模型与结果被保存
- 可以加载历史模型或历史实验结果
- 可以选择测试集并查看测试结果
- 可以展示至少一张结果图

## 11. 当前不建议优先做的内容

为了控制工作量，以下内容不建议放在第一优先级：

- 原始流量文件上传
- 在线实时检测
- 抓包解析
- 用户登录与权限管理
- 数据库存储
- 复杂网络部署

你当前的重点仍然是：

- 把训练、测试、结果展示链路做完整
- 让它看起来像一个本地软件系统

## 12. 当前结论

现在的界面开发方向已经从“网页式原型”切换为“本地桌面应用式原型”。

最终目标很明确：

- 通过 Electron 桌面界面组织现有联邦学习系统
- 把本地数据目录、训练流程、模型保存、测试结果整合到统一窗口中
- 满足导师对“可视化界面 + 可操作流程 + 结果展示”的要求

这将是你毕业设计最终系统最合适的实现形态之一。

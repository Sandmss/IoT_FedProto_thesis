# 原型系统完善计划

## 目标

围绕任务书要求，在当前已有代码和实验基础上，完成一个可运行、可复现、可展示结果的“基于轻量化联邦学习的恶意流量检测系统”原型。

本计划不再以继续扩展大量新算法为主，而以“系统收口”和“原型交付”为主。

## 原型系统完成标准

系统完成时应至少满足以下条件：

- 能加载已经处理好的 IoT 流量数据
- 能运行 `Local`、`FedAvg`、`FedProto` 三种训练模式
- 能运行 `IoT_MLP`、`IoT_CNN1D`、`IoT_Transformer1D` 三类模型
- 能运行至少一组异构实验，例如 `IoT_MIX_MLP_CNN1D + FedProto`
- 能输出 Accuracy、AUC、F1、FNR、FPR、训练耗时、通信量等结果
- 能通过固定入口脚本复现实验

## 当前系统基础

你当前已经具备以下核心基础：

- 数据预处理脚本与客户端划分脚本已经存在
- `src/main.py` 已经是统一主入口
- 已实现 `Local`、`FedAvg`、`FedProto`
- 已实现 `IoT_MLP`、`IoT_CNN1D`、`IoT_Transformer1D`
- 已补充 `IoT_MIX_MLP_CNN1D` 异构模型入口
- 已有多组实验结果与 `.h5` 输出

因此，接下来重点不是从零开发，而是把这些内容整理成完整原型系统。

## 系统模块划分

建议把整个原型系统分成 5 个模块来收口：

### 1. 数据处理模块

职责：

- 读取原始流量特征数据
- 执行清洗、归一化、采样
- 生成客户端训练/测试集

需要保证：

- 数据集路径清晰
- 不同数据划分方式可复现
- `split_stats.json` 可作为划分依据输出

### 2. 本地模型模块

职责：

- 提供轻量化本地检测模型
- 支持统一的 `extract_features()` 接口
- 保证不同模型的特征维度可对齐

当前主模型：

- `IoT_MLP`
- `IoT_CNN1D`
- `IoT_Transformer1D`
- `IoT_MIX_MLP_CNN1D`

### 3. 联邦训练模块

职责：

- 提供 `Local`
- 提供 `FedAvg`
- 提供 `FedProto`

需要保证：

- 同构实验可稳定运行
- 异构实验至少支持 `FedProto`
- 日志与结果文件正常保存

### 4. 结果评估模块

职责：

- 输出 Accuracy、AUC Macro、AUC Micro、Precision、Recall、F1、FNR、FPR
- 输出训练总耗时、平均每轮耗时、推理延迟
- 输出通信量、参数量、模型大小、FLOPs

### 5. 系统交付模块

职责：

- 统一入口
- 明确目录结构
- 提供 README
- 提供最小可运行命令

## 接下来的执行顺序

### 阶段 1：统一系统入口

任务：

- 明确 `src` 为最终系统入口
- 保留以下正式脚本：
  - `src/run_local.sh`
  - `src/run_fedavg.sh`
  - `src/run_fedproto.sh`
  - `src/run_fedproto_mix_mlp_cnn1d.sh`
- 在 README 中说明 `sysytem` 为旧目录或参考目录，不作为最终主入口

验收标准：

- 老师或评审只看 `src` 就能理解系统怎么运行

### 阶段 2：整理结果输出与目录

任务：

- 统一 `results` 目录组织方式
- 将结果按模型或实验类型清晰归档
- 单独为异构实验预留目录

建议结果目录：

- `results/MLP模型`
- `results/CNN1D模型`
- `results/transformer模型`
- `results/异构模型`

验收标准：

- 所有结果能根据目录快速追溯到算法、模型和设置

### 阶段 3：补齐运行说明

任务：

- 编写完整 README
- 写清：
  - 项目目标
  - 目录结构
  - 环境依赖
  - 数据准备方式
  - 同构实验命令
  - 异构实验命令
  - 结果文件说明

验收标准：

- 其他人拿到仓库后能按 README 跑通至少一组实验

### 阶段 4：整理总结果表

任务：

- 建立统一结果表
- 纳入三类模型的同构实验结果
- 纳入异构 `FedProto` 扩展实验结果
- 纳入轻量化统计结果

建议字段：

- 数据集
- 设置
- 模型
- 算法
- Acc
- AUC Macro
- AUC Micro
- Precision
- Recall
- F1
- FNR
- FPR
- 训练总耗时
- 平均每轮耗时
- 通信量
- 日志路径

验收标准：

- 一张总表就能看出系统整体效果和轻量化优势

### 阶段 5：补系统展示材料

任务：

- 画系统结构图
- 画 FedProto 流程图
- 整理关键结果图

建议图示内容：

- 客户端本地训练
- 特征提取
- 原型上传
- 服务端聚合
- 原型下发
- 评估与结果保存

验收标准：

- 系统不仅能跑，还能被清楚展示和解释

## 当前最优先做的事

在异构实验运行的同时，建议立刻并行推进下面三件事：

1. 完成 README
2. 整理总结果表
3. 规划异构实验结果目录

## 当前不建议优先做的事

- 不优先继续新增很多模型
- 不优先继续做大量超参数消融
- 不优先正式实现剪枝和量化
- 不优先做复杂 GUI 界面

这些都可能有价值，但不影响原型系统是否成立。

## 一句话结论

你当前已经不是“缺核心算法”，而是进入“原型系统收口阶段”。  
接下来最应该做的是把已有的模型、联邦训练流程、异构支持、结果评估和运行说明统一成一个完整可交付系统。
